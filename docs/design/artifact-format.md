# Design invariant: compiled artifacts are version-locked

**Audience:** anyone editing parseman's compiler, linker, or macro plugin — especially
the compiled-artifact format (`LinkablePieces`, `firstSetRecipes`, the fused-body
shape, the serialized `rules()`/`composeLeaf()` output a package ships).

## The rule

> **A grammar is compiled AND fused/linked by the SAME parseman version.** Parseman
> does **not** read a compiled artifact produced by a different version. The
> compiled-artifact format may therefore change **freely** between versions and
> **must not carry back-compat shims, `Legacy*` shapes, or dual-format read paths**
> for an older artifact shape. If you find yourself adding a compatibility branch for
> an older artifact format, **STOP** — it is dead code by design.

A "compiled artifact" is any parseman-generated output another step consumes: the
macro-emitted `_r_<Name>` functions and their serialized `LinkablePieces`, the
first-set recipes, the fused-body source. These are produced by the macro at build
time and consumed by `fusedBody`/`compose` in the *same* build, against the *same*
parseman. They are never carried across a version boundary — a package that ships a
compiled artifact ships one compiled by *its* pinned parseman, and a consumer that
`compose([...])`s it does so with the *same* pinned parseman (the artifact re-lowers
from its carried IR under the consumer's parseman, or is rejected).

## Why it holds (and is enforced)

- **Stamp.** Every artifact is stamped with the producing version — the generated-code
  banner (`// Generated by parseman vX.Y.Z …`) and the `v` field on `LinkablePieces`
  both come from `PARSEMAN_VERSION` (`src/version.ts`, kept in sync with `package.json`
  by `test/unit/version-sync.test.ts`).
- **Assertion.** `fusedBody` (`src/compiler/linker.ts`) refuses to link a serialized
  piece whose `v` differs from the linking parseman, failing loudly with a
  "recompile — parseman does not fuse across versions" message rather than silently
  mis-reading a stale shape.

## Cautionary example

PR #48 (`fix(codegen): cross-artifact first-set dispatch`) originally shipped a
`LegacyFirstSetRecipe` type and a `{concrete, refs}` read branch in `fusedBody` "for
back-compat" with the previous recipe shape. There was nothing to be compatible with:
no older-format artifact is ever fed to a newer parseman. It was dead defensive code
and was deleted; the ordered-chain `{alts}` recipe is the *sole* format. Do not
reintroduce that class of shim.
